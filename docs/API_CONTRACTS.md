Canonical shape of every external-facing surface. If this file drifts from code, the principal-reviewer gate fails.

## Inbound: `POST /webhooks/wati`

**Auth**: header `X-Webhook-Secret: $WATI_WEBHOOK_SECRET`.

**Idempotency**: `wati_message_id` is unique on the `turns` table; duplicates are a no-op 200.

**Expected body (WATI v1 webhook shape, subset we use):**

```jsonc
{
  "id": "wamid.HBgMOT...=",             // WATI message id — idempotency key
  "waId": "919876543210",               // E.164 without '+', the sender's WhatsApp phone
  "senderName": "Priya R.",
  "text": "find me a sales cofounder",  // present for text messages
  "type": "text" | "button" | "interactive",
  "buttonPayload": "ACCEPT" | "SKIP" | "DECLINE" | null,  // for button replies
  "interactive": { /* raw interactive payload from WATI, optional */ },
  "timestamp": 1713537600
}
```

**Response**: `200 { ok: true }` on accept or duplicate. `401` on bad secret. `400` on schema mismatch.

Handler pipeline: verify secret → parse → insert turn (on conflict do nothing) → dispatch to IntentRouter asynchronously → 200.

## Outbound: WATI client calls

All calls go to `$WATI_API_BASE_URL` with `Authorization: Bearer $WATI_API_TOKEN`.

### `sendText(waId, text)`
```
POST /api/v1/sendSessionMessage/{waId}
Body: { "messageText": "<text>" }
```

### `sendInteractiveButtons(waId, body, buttons)`
```
POST /api/v2/sendInteractiveButtonsMessage?whatsappNumber={waId}
Body: {
  "body": "<prompt text>",
  "buttons": [ { "text": "Accept" }, { "text": "Skip" } ]
}
```
Used for Accept/Skip on candidate cards and Accept/Decline on consent requests.

### `sendTemplate(waId, templateName, params)`
```
POST /api/v1/sendTemplateMessage?whatsappNumber={waId}
Body: {
  "template_name": "<templateName>",
  "broadcast_name": "cofounder_bot",
  "parameters": [ { "name": "founder_name", "value": "Priya" } ]
}
```
Only used to reopen a session outside the 24h window.

## Admin routes

All admin routes require `Authorization: Bearer $ADMIN_TOKEN`.

### `POST /admin/ingest`
Triggers CSV → Postgres ingest.

Request: `{ "csvPath": "data/seed_founders.csv" }`
Response: `{ "inserted": 120, "updated": 0, "skipped": 0 }`

### `GET /admin/stats`
Response:
```json
{
  "founders": 120,
  "conversations": 42,
  "candidates_shown": 310,
  "accepts": 27,
  "mutual_accepts": 11,
  "intros_sent": 11,
  "declined": 4,
  "expired": 2
}
```

## Internal: Matching pipeline payloads

Not exposed over HTTP; documented for cross-module contracts.

### `SearchState` (per conversation)
```ts
{
  role?: "technical" | "sales" | "growth" | "product" | "ops" | "design" | "generalist";
  sector: string[];              // e.g. ["fintech", "b2b"]
  stage: string[];               // e.g. ["pre-seed", "seed"]
  location: string[];            // normalized city/region names
  seniority?: "operator" | "founder-level" | "senior-ic";
  must_have: string[];           // free-text requirements
  nice_to_have: string[];
  anti_prefs: string[];          // from Skip signals + explicit "not X"
}
```

### `RefinementDelta` (LLM output)
```ts
{
  add: Partial<SearchState>;
  remove: Partial<SearchState>;
  anti_prefs: string[];
}
```

### `CandidateCard` (sent to WhatsApp)
```ts
{
  founder_id: string;
  name: string;
  city: string;
  headline: string;       // 1 line
  rationale: string;      // 1 line "why"
}
```

## Consent: payloads across the state machine

### Target-facing consent request (outbound WhatsApp)
```
{requesterName} ({requesterCity}) wants to connect on Build3 Cofounder Bot.

Why: {1–2 line rationale, generic — NO requester-specific profile fields beyond name/city}

Reply Accept to let us intro you, or Decline to pass.
```

### Intro message (both founders, after mutual accept)
```
Hey {A} and {B} — you both said yes to an intro via Build3 Cofounder Bot.

{one-line who-A-is} ↔ {one-line who-B-is}

Why it might work: {1–2 lines}

Over to you.
```
